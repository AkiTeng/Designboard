# 简版技术框架文档（Technical Framework Draft v0）

## 1. 文档目的 Purpose

这份文档用于在产品文档之外，单独提供一份可讨论、可评审、可继续细化的技术实现框架。

当前目标不是直接定稿，而是先明确系统边界、核心模块、基础技术选型、交付约束与风险点，帮助后续研发评审和方案收敛。

如果确定以 PixiJS 作为桌面主画布渲染方案，配套实施细化见 `docs/pixijs-development-guide.md`。

## 2. 与产品文档的关系 Relationship to PRD

本文档与 `docs/prd-draft.md` 分离维护，两者职责不同：

- 产品文档：说明目标、用户问题、范围与需求
- 技术框架文档：说明实现分层、模块职责、技术选型与工程约束

两份文档应保持一一对应，但不混写。

## 3. 设计原则 Design Principles

本阶段技术方案遵循以下原则：

1. 优先保证核心流程可实现、可验证
2. 优先采用低复杂度、易维护的实现方式
3. 保持模块边界清晰，避免过早抽象
4. 为后续扩展预留结构，但不过度设计
5. 先满足交付和稳定性，再逐步优化性能与体验
6. 画布性能属于基础架构约束，不作为后期补救项处理

## 4. 系统目标 Technical Goals

本次技术框架的目标是：

1. 明确系统的基础组成和职责划分
2. 定义前后端与数据层的基本交互关系
3. 给出适合初版交付的实现结构
4. 标记关键依赖、风险与待确认项
5. 为后续详细设计、任务拆分和开发排期提供基础

## 5. 非目标 Non-goals

当前阶段不包含：

- 细化到每个接口字段级别的 API 设计
- 完整数据库表结构定稿
- 高可用、多地域、复杂容灾设计
- 面向远期规模的过度性能优化
- 未确认业务价值的复杂扩展能力

## 6. 总体架构 Overview

基于当前第一版需求，建议采用“前端主导、本地文件持久化、多服务商 API 适配”的技术结构：

- 表现层：桌面端无限画布 + 移动端流式列表，两端共享核心生成能力
- 应用层：负责画布交互、项目与画布管理、生成任务编排、撤回删除等流程控制
- 渲染层：负责图片节点、设计稿节点、后续特效节点的统一绘制与调度
- 资源层：负责原图、缩略图、Prompt 记录、本地文件结构、回收桶管理
- 集成层：负责多个第三方生图服务商的统一适配和调用
- 性能层：负责缩略图优先渲染、视区内展示、交互流畅度优化，以及纹理预算、shader 调度与降级

第一版暂不以中心化后端为核心，而是优先把单机、本地、前端可运行的能力做稳定。
同时，画布渲染结构不能只服务于静态图片，需要为类似 Paper 的设计稿展示和后续 GPU 特效扩展预留统一节点模型。
桌面主画布不应建立在大量 DOM 节点之上，而应采用独立 runtime + Canvas/WebGL 渲染层的方式，以避免后续性能瓶颈触发整体重构。
如果目标场景包含“多张 4K 图片 + 多个 GLSL shader 同屏 + 持续缩放/平移”，则该能力必须被视为一开始就要收敛的基础架构，不应留到上线后补救。

## 7. 核心模块 Module Breakdown

### 7.1 Desktop Client / 桌面端前端

职责：

- 提供无限画布作为主工作区
- 支持图片节点、设计稿节点展示，以及统一的框选、多选、拖拽、删除、撤回
- 支持通过输入框直接发起纯文本生图或参考图生图
- 支持触控板双指缩放、拖动画布等桌面交互
- 在高密度图片场景下优先渲染缩略图，保证画布流畅性
- 第一版设计稿节点以静态展示为主，但渲染协议需要兼容后续 shader / effect 扩展

### 7.2 Mobile Client / 移动端前端

职责：

- 提供独立于桌面端的流式布局，而不是复用无限画布交互
- 以从上到下的图片列表展示生成内容，最新结果优先出现在底部
- 保留核心输入区，支持 Prompt 输入和上传参考图
- 支持点击图片进入全屏浏览
- 避免第一版引入移动端拖拽和缩放等高复杂交互

### 7.3 Project & Canvas Runtime

职责：

- 管理多个项目和多个画布
- 维护当前打开的画布状态和节点结构
- 处理图片节点、参考图状态、撤回操作和删除逻辑
- 负责打开旧画布、新建画布和在画布之间切换
- 维护统一节点协议，避免图片节点与设计稿节点分裂成两套运行时
- 维护独立于 React 组件树的画布 runtime，避免节点渲染与 UI 状态强耦合
- 维护可见区域、dirty 状态和局部刷新调度信息

### 7.4 Local Storage & File Manager

职责：

- 以本地文件夹作为持久化基础，而不是仅依赖浏览器临时存储
- 以“一个画布对应一个文件夹”的方式组织数据
- 在画布文件夹内保存原图、缩略图、Prompt 和结构化配置数据
- 管理项目级或画布级回收桶，用于存放被删除图片
- 支持重新打开本地已保存的项目或画布

建议最小保存内容包括：

- 画布结构
- 图片原图
- 缩略图
- Prompt
- 图片位置与展示状态
- 服务商与生成参数记录
- 设计稿节点配置和后续特效扩展字段

### 7.5 Image Processing Pipeline

职责：

- 接收第三方服务商返回的原始图片
- 生成用于画布显示的缩略图
- 区分原图用途和缩略图用途
- 为图片浏览、导出和画布渲染提供不同尺寸资源
- 为不同缩放级别准备可切换的资源层级，如 thumbnail、preview、original

建议要求：

- 原图保留完整质量，用于下载和大图查看
- 缩略图用于日常渲染和画布缩放场景
- 缩略图不能过度压缩，需要在性能与可辨识度之间平衡
- 资源切换应尽量异步完成，避免缩放过程中阻塞主线程

### 7.6 AI Provider Adapter / 第三方服务适配层

职责：

- 为多个第三方服务商提供统一调用接口
- 支持用户在设置页中分别填写各服务商 API Key
- 统一处理文本生图、参考图生图、多图返回等能力差异
- 对不同服务商的入参与出参做映射和标准化

建议要求：

- UI 层不直接感知不同服务商的细节差异
- 前端使用统一生成协议，如 prompt、referenceImages、count、providerId 等
- 服务商差异尽量收敛在适配层内部

### 7.7 Canvas Rendering Engine / 画布渲染引擎

职责：

- 提供统一节点渲染入口，至少覆盖 image、designDraft 两类节点
- 将节点布局、可见性裁剪、缩放级别和命中检测统一调度
- 为后续 effect 节点或 designDraft 节点上的 shader layer 预留扩展接口
- 控制渲染降级策略，避免特效能力影响第一版主路径稳定性

建议要求：

- 第一版只要求稳定展示静态内容，不要求上线实时 GLSL 编辑
- 节点协议中预留 `renderMode`、`effectConfig`、`assets` 等扩展字段
- 渲染引擎要支持按节点类型分发 renderer，而不是把所有逻辑写死在图片组件中
- 渲染引擎需要支持视区裁剪、分层渲染和局部重绘，而不是每次交互都全量重绘整张画布
- designDraft 节点应优先采用离屏缓存或纹理缓存，避免每帧重复进行复杂布局计算
- 图片、designDraft、effect 至少拆成独立渲染层，避免每次 effect 刷新都污染整张场景
- shader 节点必须支持 render-to-texture，复杂效果不应直接在主画布逐节点串行重算
- 缩放和平移过程中应优先复用已有纹理和离屏缓存，停止交互后再异步补高质量资源

### 7.8 Performance Infrastructure / 性能基础设施

职责：

- 维护 scene graph，用于统一管理节点可见性、层级、包围盒和 dirty 状态
- 提供 viewport culling，只渲染当前视区内或临近视区的节点
- 提供空间索引，用于命中检测、框选和批量节点查询
- 提供 asset manager，统一管理纹理加载、缓存、释放和资源等级切换
- 提供降级策略，在 effect 不可用或资源紧张时优先保证主画布交互流畅

建议要求：

- 第一版即接入基础空间索引，不依赖全量数组遍历做 hit test
- 第一版即接入多级资源策略，避免原图直接参与日常画布渲染
- React 只负责应用壳层和面板，不直接承担主画布节点渲染
- 拖拽、缩放、框选应优先触发局部刷新，而不是全量重绘
- 图片解码、缩略图生成和资源升级应尽量放到 Worker / 原生侧，避免阻塞主线程交互
- asset manager 需要维护显存预算、纹理引用计数、最近最少使用淘汰和资源分级加载
- 需要区分“交互态”和“静止态”渲染策略：交互态保帧率，静止态补清晰度

## 8. 性能边界与架构收敛 Performance Envelope

如果要支撑“大量 4K 图片 + GLSL shader 同屏 + 高频缩放/平移”，技术框架需要明确以下现实约束。

### 8.1 基本判断

- 仅从当前文档的方向看，架构方向是对的，但还不能证明一定扛得住该场景
- 如果按当前草稿直接实现，能较稳地覆盖“中等数量图片 + 少量特效节点”的第一版
- 如果没有纹理预算、资源分级、shader 隔离和离屏缓存，多张 4K 图同时上屏时大概率会先撞显存和主线程抖动
- 结论上，当前框架“可以扩展到该目标”，但“当前文档定义还不足以保证该目标”

### 8.2 为什么 4K 图片是硬约束

- 单张 4K RGBA 纹理理论占用约 `3840 x 2160 x 4`，约 31.6 MB，未计入 mipmap、双缓冲和离屏纹理
- 如果同屏存在 20 张 4K 图，即使只保留一份 GPU 纹理，也可能接近或超过 600 MB 显存占用
- shader 效果通常还会引入额外 framebuffer、pass 纹理和中间结果，实际成本会继续上升
- 因此，第一版不能把“原图直接上 GPU”当成常态路径，必须以 preview / thumbnail 为默认渲染资源

### 8.3 建议的性能目标

- 交互目标：桌面端在中型画布下缩放、平移、框选保持 55-60 FPS
- 降级目标：重负载场景下允许短时降到 30 FPS，但不能出现持续性卡死或秒级掉帧
- 资源目标：默认只让视区内和临近视区节点持有高等级纹理
- 恢复目标：停止交互后 150-400 ms 内开始补高质量资源，分批完成，不抢主线程
- 容量目标：首版需要先定义一个可验收基线，例如“同屏 100-200 个节点，其中 20-40 个可见图片节点，少量 shader 节点”

### 8.4 第一版必须落地的能力

- WebGL 作为桌面主画布基础，不以 DOM 或 2D canvas 作为主渲染路径
- 多级资源体系：`thumbnail -> preview -> original`，默认渲染 preview，不直接使用 original
- viewport culling 和 near-viewport preload，只处理当前视区与边缘缓冲区
- 纹理预算与淘汰机制，达到阈值后先释放高等级纹理，再降级 effect
- shader 节点隔离：复杂特效走离屏 pass，并支持按节点或按图层单独关闭
- 交互态降级：缩放/平移时暂停非关键 shader 动画、暂停高成本 pass、锁定到低等级纹理
- 静止态回补：交互结束后再恢复 shader 与高质量纹理
- 可观测性：必须记录帧率、主线程耗时、纹理数、显存估算、离屏 pass 数量

### 8.5 技术选型收敛建议

- 如果画布以 2D 节点编排为主，优先评估 PixiJS + 自定义 renderer / filter 体系
- 如果 shader 表达和后处理链会很重，需要尽早评估基于 Three.js / regl /自研 WebGL pipeline 的成本
- 不建议第一版同时追求“复杂实时 shader 编辑器”和“稳定生产画布”，应先把 shader 作为受控的 effect runtime
- 如果 designDraft 本身有复杂排版和多层混合，建议优先生成静态快照或离屏缓存，而不是每帧重新排版

## 9. 基础技术选型建议 Suggested Stack

### 前端 Frontend

第一版建议以前端为核心实现：

- 宿主形态：当前以浏览器运行作为第一优先，后续支持以 SwiftUI + WKWebView 封装 macOS App
- 宿主封装原则：浏览器、Tauri、SwiftUI + WKWebView 都应被视为可替换宿主，宿主能力必须通过统一 bridge / adapter 收口
- Web 应用：React / Next.js 或同类成熟前端框架
- 桌面端画布：使用 Canvas / WebGL 类渲染方案，避免大量节点时纯 DOM 成本过高
- 推荐优先评估 PixiJS 一类成熟 2D GPU 渲染层，而不是直接把无限画布建立在 DOM 组件树上
- 节点渲染：采用“统一场景层 + 节点 renderer 注册表”模式，便于接入 designDraft 与 effect renderer
- 移动端界面：独立布局，不强行复用桌面画布交互
- 状态管理：按项目、画布、节点、设置分层管理本地状态
- 画布状态建议使用独立 store 或 runtime 容器，不让 React render cycle 决定主画布刷新节奏
- 图片解码优先使用 `createImageBitmap`、Worker 或宿主侧能力，避免在 React 生命周期中同步解码大图
- 如果目标环境允许，需预研 OffscreenCanvas 可用性；如果不可用，也要保证渲染循环与资源解码解耦

### 本地存储 Local Persistence

建议优先使用本地文件系统能力，而不是只依赖浏览器缓存：

- 一个项目或画布对应一个本地目录
- 目录内保存结构化 JSON 数据、原图资源、缩略图资源和回收桶内容
- 支持关闭应用后重新打开本地项目
- 设计稿节点引用的静态资源、样式快照或序列化布局数据也需要进入同一持久化体系

### 第三方集成 Integrations

- 支持多个图片生成服务商
- 每个服务商独立保存 API Key
- 用户填写 API Key 后即可启用对应服务商
- 不在第一版暴露复杂的服务商差异化 UI

### 工程化 Engineering

- 版本管理：Git
- 构建与测试：至少包含 lint、build 和基础交互验证
- 配置管理：本地配置 + 服务商设置
- 第一版优先保证桌面端主路径，再保证移动端核心能力
- 对 designDraft / effect 扩展位做 schema 校验，避免未来节点协议失控
- 性能回归需要纳入基础验证，至少跟踪中型画布打开、拖拽和缩放时的基线数据
- 宿主能力如文件系统、系统菜单、窗口与原生对话框需通过 host adapter / bridge 收口，避免任何宿主 API 直接渗透到 React 组件和 editor runtime

## 10. 关键流程建议 Key Flows

建议重点围绕以下主路径设计：

### 流程 A：纯文本生图

1. 用户在桌面端或移动端输入 Prompt
2. 用户选择一个已配置 API Key 的服务商
3. 系统发起生成请求
4. 返回原图后生成缩略图
5. 将结果写入本地文件结构
6. 桌面端将缩略图放回画布，移动端将结果追加到列表底部

### 流程 B：参考图生图

1. 用户通过输入框上传参考图，或在桌面画布中选中已有图片
2. 输入框中显示当前参考图缩略图
3. 用户输入 Prompt 并发起生成
4. 系统将参考图和文本参数交给统一适配层
5. 返回结果后生成缩略图并写入本地存储
6. 结果回填到当前上下文中继续迭代

### 流程 C：打开本地项目 / 画布

1. 用户选择新建项目或打开本地已有项目
2. 系统读取本地文件夹中的结构化数据
3. 恢复项目列表、画布列表、图片节点、缩略图索引和回收桶状态
4. 桌面端恢复为无限画布，移动端恢复为流式列表展示

### 流程 D：删除与回收桶

1. 用户删除画布中的图片
2. 系统更新画布结构数据
3. 被删除图片移动到回收桶目录，而不是立即物理删除
4. 如后续需要，可基于回收桶扩展恢复能力

### 流程 E：设计稿节点展示

1. 用户在桌面画布中创建或导入一个设计稿节点
2. 系统将布局、文本、图片引用和样式信息序列化为统一节点结构
3. 渲染引擎根据 designDraft renderer 将内容绘制到画布
4. 缩放、拖拽和选中行为继续复用统一画布交互能力
5. 保存项目时同步写入节点配置和关联资源

### 流程 F：画布渲染与资源切换

1. 用户打开画布后，runtime 先恢复节点数据和基础 scene graph
2. renderer 根据 viewport 只装载当前视区内节点的低成本资源
3. 用户缩放或移动视图时，系统先保持交互流畅，再异步切换更高等级资源
4. 节点内容发生变化时，仅标记对应 dirty 区域并触发局部刷新
5. 当节点离开视区或长时间未使用时，asset manager 释放高成本纹理缓存

## 11. 数据与状态设计原则 Data and State

建议围绕以下核心实体建模：

- Project：项目
- Canvas：画布
- CanvasNode：统一画布节点
- ImageAsset：图片资源
- PromptRecord：生成记录
- ProviderConfig：服务商配置
- TrashItem：回收桶条目

每个 ImageAsset 建议至少包含以下信息：

- 原图路径
- 缩略图路径
- Prompt
- 生成时间
- 所属项目 / 画布
- 图片尺寸
- 在桌面画布中的位置与层级
- 来源服务商与生成参数
- 当前是否被删除或移入回收桶
- 多级资源路径，如 thumbnail、preview、original
- 资源尺寸、文件大小和估算纹理占用

每个 CanvasNode 建议至少包含以下信息：

- 节点类型，如 image、designDraft、effect
- 基础布局信息，如 x、y、width、height、zIndex
- 渲染模式，如 static、gpuEffect
- 关联资源列表
- 节点私有配置，如 design tokens、样式片段、effectConfig
- 可选的交互元数据，如是否可选中、是否锁定、是否隐藏

运行时建议额外维护 SceneNode，但不直接作为持久化对象：

- 节点包围盒
- 当前是否在视区内
- 当前 dirty 状态
- 当前 renderer 实例引用
- 当前使用的资源等级，如 thumbnail、preview、original
- 当前纹理句柄、最后访问时间和预估显存成本
- 当前是否处于交互降级态

数据设计原则：

- 原图与缩略图必须分离存储
- 画布节点模型必须独立于具体渲染实现
- 画布结构与资源文件路径必须解耦
- Prompt 与图片关系必须可追溯
- 删除逻辑优先采用软删除到回收桶
- 单项目、多画布结构需要支持独立打开和恢复
- 第一版即使不启用 GLSL，也要保证 effect 配置字段能被安全保存和忽略
- 运行时状态与持久化状态必须分离，不能把 renderer 实例或缓存信息直接写入项目文件

## 12. 安全与稳定性 Security and Reliability

第一版建议至少包含：

- API Key 本地保存时需要避免明文四处散落，至少集中管理
- 上传图片、参考图和生成结果需做基础格式校验
- 第三方服务调用需有超时、失败和重试提示
- 同一次生成流程要能区分“请求中、成功、失败”状态
- 删除动作和写文件动作要尽量可恢复，避免误删后数据直接丢失
- 对未启用或当前版本不支持的 effect 配置，需要具备安全降级与兼容读取策略
- 对 shader 编译失败、纹理分配失败、上下文丢失需要具备恢复或降级策略

## 13. 可观测性 Observability

虽然第一版以前端为主，但仍建议保留基础问题定位能力：

- 本地错误日志
- 服务商调用失败原因记录
- 关键操作事件记录，如生成、删除、打开项目、切换画布
- 基础性能指标，如首屏加载、缩放流畅度、批量图片展示性能
- 设计稿节点渲染耗时和未来 effect 渲染失败原因需要可追踪
- 建议补充纹理内存占用、视区节点数、资源切换耗时等指标
- 建议补充 shader 编译耗时、offscreen pass 次数、context lost 次数和交互态/静止态切换频率

## 14. 部署与环境 Deployment

第一版建议以本地前端运行环境为主：

- 本地开发环境
- 本地构建产物
- 如后续需要，再考虑封装为桌面客户端或接入独立后端

当前阶段不以多环境部署、账号系统或中心化云服务为前提

## 15. 依赖 Dependencies

当前技术框架依赖以下信息进一步明确：

- 产品侧确认核心业务流程
- 设计侧确认关键交互与页面结构
- 设计侧需补充类似 Paper 的目标展示形态和可接受的第一版降级范围
- 研发侧确认团队熟悉技术栈
- 运维侧确认部署平台与资源限制
- 如涉及第三方服务，需确认接入方式、额度与 SLA

## 16. 风险 Risks

当前主要风险包括：

- 业务定义不充分，导致技术设计偏空泛
- 过早细化实现，后续返工成本高
- 外部依赖不明确，影响交付边界
- 初版职责划分不清，导致开发并行效率低
- 缺少验收口径，导致“开发完成”与“可交付”认知不一致
- 如果第一版节点模型只适配图片，后续接设计稿和 shader 效果会造成结构性返工
- 如果设计稿展示直接绑在 DOM 组件树上，后续统一缩放、裁剪和 GPU 效果接入会很困难
- 如果没有 asset manager 和多级资源策略，图片与设计稿混排后容易出现显存和内存失控
- 如果没有 viewport culling 和空间索引，节点规模上来后命中检测与拖拽体验会明显劣化
- 如果 shader 节点缺少离屏缓存和分层调度，多 effect 同屏时会直接拖垮主画布帧率
- 如果没有明确性能基线，团队会误以为“能显示出来”就等于“可交付”

## 17. 验收标准 Acceptance Criteria

满足以下条件，可视为技术框架初稿可进入下一轮评审：

- 已与产品文档分离成独立文件
- 已定义系统基础分层和核心模块
- 已明确建议技术栈方向
- 已描述主流程的基本处理路径
- 已识别主要依赖与风险
- 已明确设计稿节点与后续 effect 扩展的技术预留方式
- 已明确主画布不以 DOM 为核心渲染载体，并定义基础性能机制
- 已给出第一版需要具备的 scene graph、asset manager、viewport culling 等能力方向
- 已明确 4K 图片、shader 节点、缩放/平移场景下的性能边界与降级策略
- 文档能够支撑后续接口设计、任务拆分和工期评估
