# 简版技术框架文档（Technical Framework Draft v0）

## 1. 文档目的 Purpose

这份文档用于在产品文档之外，单独提供一份可讨论、可评审、可继续细化的技术实现框架。

当前目标不是直接定稿，而是先明确系统边界、核心模块、基础技术选型、交付约束与风险点，帮助后续研发评审和方案收敛。

## 2. 与产品文档的关系 Relationship to PRD

本文档与 `docs/prd-draft.md` 分离维护，两者职责不同：

- 产品文档：说明目标、用户问题、范围与需求
- 技术框架文档：说明实现分层、模块职责、技术选型与工程约束

两份文档应保持一一对应，但不混写。

## 3. 设计原则 Design Principles

本阶段技术方案遵循以下原则：

1. 优先保证核心流程可实现、可验证
2. 优先采用低复杂度、易维护的实现方式
3. 保持模块边界清晰，避免过早抽象
4. 为后续扩展预留结构，但不过度设计
5. 先满足交付和稳定性，再逐步优化性能与体验

## 4. 系统目标 Technical Goals

本次技术框架的目标是：

1. 明确系统的基础组成和职责划分
2. 定义前后端与数据层的基本交互关系
3. 给出适合初版交付的实现结构
4. 标记关键依赖、风险与待确认项
5. 为后续详细设计、任务拆分和开发排期提供基础

## 5. 非目标 Non-goals

当前阶段不包含：

- 细化到每个接口字段级别的 API 设计
- 完整数据库表结构定稿
- 高可用、多地域、复杂容灾设计
- 面向远期规模的过度性能优化
- 未确认业务价值的复杂扩展能力

## 6. 总体架构 Overview

基于当前第一版需求，建议采用“前端主导、本地文件持久化、多服务商 API 适配”的技术结构：

- 表现层：桌面端无限画布 + 移动端流式列表，两端共享核心生成能力
- 应用层：负责画布交互、项目与画布管理、生成任务编排、撤回删除等流程控制
- 资源层：负责原图、缩略图、Prompt 记录、本地文件结构、回收桶管理
- 集成层：负责多个第三方生图服务商的统一适配和调用
- 性能层：负责缩略图优先渲染、视区内展示、交互流畅度优化

第一版暂不以中心化后端为核心，而是优先把单机、本地、前端可运行的能力做稳定。

## 7. 核心模块 Module Breakdown

### 7.1 Desktop Client / 桌面端前端

职责：

- 提供无限画布作为主工作区
- 支持图片节点展示、框选、多选、拖拽、删除、撤回
- 支持通过输入框直接发起纯文本生图或参考图生图
- 支持触控板双指缩放、拖动画布等桌面交互
- 在高密度图片场景下优先渲染缩略图，保证画布流畅性

### 7.2 Mobile Client / 移动端前端

职责：

- 提供独立于桌面端的流式布局，而不是复用无限画布交互
- 以从上到下的图片列表展示生成内容，最新结果优先出现在底部
- 保留核心输入区，支持 Prompt 输入和上传参考图
- 支持点击图片进入全屏浏览
- 避免第一版引入移动端拖拽和缩放等高复杂交互

### 7.3 Project & Canvas Runtime

职责：

- 管理多个项目和多个画布
- 维护当前打开的画布状态和节点结构
- 处理图片节点、参考图状态、撤回操作和删除逻辑
- 负责打开旧画布、新建画布和在画布之间切换

### 7.4 Local Storage & File Manager

职责：

- 以本地文件夹作为持久化基础，而不是仅依赖浏览器临时存储
- 以“一个画布对应一个文件夹”的方式组织数据
- 在画布文件夹内保存原图、缩略图、Prompt 和结构化配置数据
- 管理项目级或画布级回收桶，用于存放被删除图片
- 支持重新打开本地已保存的项目或画布

建议最小保存内容包括：

- 画布结构
- 图片原图
- 缩略图
- Prompt
- 图片位置与展示状态
- 服务商与生成参数记录

### 7.5 Image Processing Pipeline

职责：

- 接收第三方服务商返回的原始图片
- 生成用于画布显示的缩略图
- 区分原图用途和缩略图用途
- 为图片浏览、导出和画布渲染提供不同尺寸资源

建议要求：

- 原图保留完整质量，用于下载和大图查看
- 缩略图用于日常渲染和画布缩放场景
- 缩略图不能过度压缩，需要在性能与可辨识度之间平衡

### 7.6 AI Provider Adapter / 第三方服务适配层

职责：

- 为多个第三方服务商提供统一调用接口
- 支持用户在设置页中分别填写各服务商 API Key
- 统一处理文本生图、参考图生图、多图返回等能力差异
- 对不同服务商的入参与出参做映射和标准化

建议要求：

- UI 层不直接感知不同服务商的细节差异
- 前端使用统一生成协议，如 prompt、referenceImages、count、providerId 等
- 服务商差异尽量收敛在适配层内部

## 8. 基础技术选型建议 Suggested Stack

### 前端 Frontend

第一版建议以前端为核心实现：

- Web 应用：React / Next.js 或同类成熟前端框架
- 桌面端画布：使用 Canvas / WebGL 类渲染方案，避免大量节点时纯 DOM 成本过高
- 移动端界面：独立布局，不强行复用桌面画布交互
- 状态管理：按项目、画布、节点、设置分层管理本地状态

### 本地存储 Local Persistence

建议优先使用本地文件系统能力，而不是只依赖浏览器缓存：

- 一个项目或画布对应一个本地目录
- 目录内保存结构化 JSON 数据、原图资源、缩略图资源和回收桶内容
- 支持关闭应用后重新打开本地项目

### 第三方集成 Integrations

- 支持多个图片生成服务商
- 每个服务商独立保存 API Key
- 用户填写 API Key 后即可启用对应服务商
- 不在第一版暴露复杂的服务商差异化 UI

### 工程化 Engineering

- 版本管理：Git
- 构建与测试：至少包含 lint、build 和基础交互验证
- 配置管理：本地配置 + 服务商设置
- 第一版优先保证桌面端主路径，再保证移动端核心能力

## 9. 关键流程建议 Key Flows

建议重点围绕以下主路径设计：

### 流程 A：纯文本生图

1. 用户在桌面端或移动端输入 Prompt
2. 用户选择一个已配置 API Key 的服务商
3. 系统发起生成请求
4. 返回原图后生成缩略图
5. 将结果写入本地文件结构
6. 桌面端将缩略图放回画布，移动端将结果追加到列表底部

### 流程 B：参考图生图

1. 用户通过输入框上传参考图，或在桌面画布中选中已有图片
2. 输入框中显示当前参考图缩略图
3. 用户输入 Prompt 并发起生成
4. 系统将参考图和文本参数交给统一适配层
5. 返回结果后生成缩略图并写入本地存储
6. 结果回填到当前上下文中继续迭代

### 流程 C：打开本地项目 / 画布

1. 用户选择新建项目或打开本地已有项目
2. 系统读取本地文件夹中的结构化数据
3. 恢复项目列表、画布列表、图片节点、缩略图索引和回收桶状态
4. 桌面端恢复为无限画布，移动端恢复为流式列表展示

### 流程 D：删除与回收桶

1. 用户删除画布中的图片
2. 系统更新画布结构数据
3. 被删除图片移动到回收桶目录，而不是立即物理删除
4. 如后续需要，可基于回收桶扩展恢复能力

## 10. 数据与状态设计原则 Data and State

建议围绕以下核心实体建模：

- Project：项目
- Canvas：画布
- ImageAsset：图片资源
- PromptRecord：生成记录
- ProviderConfig：服务商配置
- TrashItem：回收桶条目

每个 ImageAsset 建议至少包含以下信息：

- 原图路径
- 缩略图路径
- Prompt
- 生成时间
- 所属项目 / 画布
- 图片尺寸
- 在桌面画布中的位置与层级
- 来源服务商与生成参数
- 当前是否被删除或移入回收桶

数据设计原则：

- 原图与缩略图必须分离存储
- 画布结构与资源文件路径必须解耦
- Prompt 与图片关系必须可追溯
- 删除逻辑优先采用软删除到回收桶
- 单项目、多画布结构需要支持独立打开和恢复

## 11. 安全与稳定性 Security and Reliability

第一版建议至少包含：

- API Key 本地保存时需要避免明文四处散落，至少集中管理
- 上传图片、参考图和生成结果需做基础格式校验
- 第三方服务调用需有超时、失败和重试提示
- 同一次生成流程要能区分“请求中、成功、失败”状态
- 删除动作和写文件动作要尽量可恢复，避免误删后数据直接丢失

## 12. 可观测性 Observability

虽然第一版以前端为主，但仍建议保留基础问题定位能力：

- 本地错误日志
- 服务商调用失败原因记录
- 关键操作事件记录，如生成、删除、打开项目、切换画布
- 基础性能指标，如首屏加载、缩放流畅度、批量图片展示性能

## 13. 部署与环境 Deployment

第一版建议以本地前端运行环境为主：

- 本地开发环境
- 本地构建产物
- 如后续需要，再考虑封装为桌面客户端或接入独立后端

当前阶段不以多环境部署、账号系统或中心化云服务为前提

## 14. 依赖 Dependencies

当前技术框架依赖以下信息进一步明确：

- 产品侧确认核心业务流程
- 设计侧确认关键交互与页面结构
- 研发侧确认团队熟悉技术栈
- 运维侧确认部署平台与资源限制
- 如涉及第三方服务，需确认接入方式、额度与 SLA

## 15. 风险 Risks

当前主要风险包括：

- 业务定义不充分，导致技术设计偏空泛
- 过早细化实现，后续返工成本高
- 外部依赖不明确，影响交付边界
- 初版职责划分不清，导致开发并行效率低
- 缺少验收口径，导致“开发完成”与“可交付”认知不一致

## 16. 验收标准 Acceptance Criteria

满足以下条件，可视为技术框架初稿可进入下一轮评审：

- 已与产品文档分离成独立文件
- 已定义系统基础分层和核心模块
- 已明确建议技术栈方向
- 已描述主流程的基本处理路径
- 已识别主要依赖与风险
- 文档能够支撑后续接口设计、任务拆分和工期评估

## 17. 待确认问题 Open Questions

1. 具体产品形态是 Web、移动端，还是多端同时支持？

3. 核心业务实体和状态流转是什么？
