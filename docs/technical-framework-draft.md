# 简版技术框架文档（Technical Framework Draft v0）

## 1. 文档目的 Purpose

这份文档用于在产品文档之外，单独提供一份可讨论、可评审、可继续细化的技术实现框架。

当前目标不是直接定稿，而是先明确系统边界、核心模块、基础技术选型、交付约束与风险点，帮助后续研发评审和方案收敛。

## 2. 与产品文档的关系 Relationship to PRD

本文档与 `docs/prd-draft.md` 分离维护，两者职责不同：

- 产品文档：说明目标、用户问题、范围与需求
- 技术框架文档：说明实现分层、模块职责、技术选型与工程约束

两份文档应保持一一对应，但不混写。

## 3. 设计原则 Design Principles

本阶段技术方案遵循以下原则：

1. 优先保证核心流程可实现、可验证
2. 优先采用低复杂度、易维护的实现方式
3. 保持模块边界清晰，避免过早抽象
4. 为后续扩展预留结构，但不过度设计
5. 先满足交付和稳定性，再逐步优化性能与体验

## 4. 系统目标 Technical Goals

本次技术框架的目标是：

1. 明确系统的基础组成和职责划分
2. 定义前后端与数据层的基本交互关系
3. 给出适合初版交付的实现结构
4. 标记关键依赖、风险与待确认项
5. 为后续详细设计、任务拆分和开发排期提供基础

## 5. 非目标 Non-goals

当前阶段不包含：

- 细化到每个接口字段级别的 API 设计
- 完整数据库表结构定稿
- 高可用、多地域、复杂容灾设计
- 面向远期规模的过度性能优化
- 未确认业务价值的复杂扩展能力

## 6. 总体架构 Overview

建议采用分层技术结构：

- 表现层：承接用户交互、状态展示与输入输出
- 应用层：承接业务流程编排、权限校验、状态流转
- 数据层：负责数据持久化、查询与基础一致性保障
- 集成层：负责调用第三方服务、消息能力或外部依赖
- 运维层：负责部署、日志、监控、告警与基础配置管理

该结构适合当前信息仍不完整的阶段，能够先建立稳定骨架，再按业务细节逐步落地。

## 7. 核心模块 Module Breakdown

### 7.1 Client / Frontend

职责：

- 提供用户操作入口和结果展示
- 管理基础页面状态和表单交互
- 对关键操作给出明确反馈
- 在异常情况下暴露可理解的错误信息

### 7.2 API / Backend Service

职责：

- 承接业务请求入口
- 处理核心业务规则
- 统一鉴权、参数校验和错误返回
- 协调数据读写与外部服务调用

### 7.3 Data Layer

职责：

- 保存核心业务数据
- 支撑基础查询与状态变更
- 提供审计、回溯或最小化历史记录能力（按业务需要）

### 7.4 External Integrations

可能包括：

- 第三方登录或身份体系
- 消息通知服务
- 文件存储服务
- 数据分析或埋点服务
- 外部业务 API

## 8. 基础技术选型建议 Suggested Stack

在未给定明确产品约束前，建议采用成熟、低认知负担的常规技术栈：

### 前端 Frontend

- Web 应用：React / Next.js 或同类成熟框架
- 样式方案：CSS Modules、Tailwind CSS 或团队已有标准
- 状态管理：优先本地状态 + 请求缓存，避免过早引入重型全局状态

### 后端 Backend

- Node.js（如 NestJS / Express）或团队主力后端技术栈
- 按 RESTful API 为主，必要时补充异步任务机制
- 对外接口保持统一错误码和响应结构

### 数据存储 Data

- 关系型数据库优先，如 PostgreSQL / MySQL
- 缓存按需引入 Redis，不作为初版必选项
- 文件内容使用对象存储或托管存储服务

### 工程化 Engineering

- 版本管理：Git
- 环境隔离：dev / staging / prod
- CI：至少包含 lint、test、build
- 配置管理：环境变量 + 明确的配置分层

## 9. 关键流程建议 Key Flows

建议至少围绕以下流程做技术设计：

1. 用户发起核心操作
2. 前端完成参数收集与基本校验
3. 后端执行鉴权、业务校验和状态变更
4. 数据层完成持久化
5. 系统返回结果并记录关键日志
6. 如涉及异步动作，通过任务队列或回调补充处理

重点不是一次覆盖所有边缘场景，而是先保证主路径闭环。

## 10. 数据与状态设计原则 Data and State

建议明确以下内容：

- 核心实体有哪些
- 每个实体的主状态是什么
- 状态如何流转
- 哪些状态变化需要记录审计日志
- 哪些数据需要幂等处理或防重复提交

## 11. 安全与稳定性 Security and Reliability

初版建议最低包含：

- 基础身份校验与权限控制
- 输入参数校验
- 敏感信息不落日志
- 关键写操作具备幂等或防重复机制
- 错误日志与请求链路可追踪
- 对外部依赖设置超时与失败处理

## 12. 可观测性 Observability

建议基础配置包括：

- 应用日志
- 错误日志聚合
- 请求耗时监控
- 核心业务事件埋点
- 发布后基础告警

## 13. 部署与环境 Deployment

建议采用至少三套环境：

- 本地开发环境
- 测试 / 预发环境
- 生产环境

## 14. 依赖 Dependencies

当前技术框架依赖以下信息进一步明确：

- 产品侧确认核心业务流程
- 设计侧确认关键交互与页面结构
- 研发侧确认团队熟悉技术栈
- 运维侧确认部署平台与资源限制
- 如涉及第三方服务，需确认接入方式、额度与 SLA

## 15. 风险 Risks

当前主要风险包括：

- 业务定义不充分，导致技术设计偏空泛
- 过早细化实现，后续返工成本高
- 外部依赖不明确，影响交付边界
- 初版职责划分不清，导致开发并行效率低
- 缺少验收口径，导致“开发完成”与“可交付”认知不一致

## 16. 验收标准 Acceptance Criteria

满足以下条件，可视为技术框架初稿可进入下一轮评审：

- 已与产品文档分离成独立文件
- 已定义系统基础分层和核心模块
- 已明确建议技术栈方向
- 已描述主流程的基本处理路径
- 已识别主要依赖与风险
- 文档能够支撑后续接口设计、任务拆分和工期评估

## 17. 待确认问题 Open Questions

1. 具体产品形态是 Web、移动端，还是多端同时支持？
2. 是否已有既定技术栈或必须复用的基础设施？
3. 核心业务实体和状态流转是什么？
4. 是否存在强实时性、高并发或高安全合规要求？
5. 这次交付的最小可上线范围是什么？
