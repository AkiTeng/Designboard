# 简版产品文档（PRD Draft v0）

## 1. 背景 Background

当前需要先产出一份简单的产品文档，作为后续 review 和迭代修改的基础版本。

这份文档的目标不是一次性定稿，而是先明确产品目标、用户问题、范围边界和评审基础，帮助后续产品、设计、研发在同一上下文里沟通。

## 2. 问题定义 Problem Statement

目前缺少一份结构化的产品定义文档，容易导致以下问题：

- 团队对产品目标理解不一致
- 用户问题和使用场景没有被明确表达
- 功能范围不清，容易不断扩张
- 后续评审缺少统一基线

## 3. 目标 Goals

本次文档的目标是：

1. 明确产品要解决的核心问题
2. 给出基础需求结构，便于后续 review
3. 明确本阶段做什么、不做什么
4. 为正式 PRD 提供初始版本

## 4. 非目标 Non-goals

当前阶段不包含：

- 详细技术实现方案
- 完整 UI 设计稿
- 长期 roadmap
- 未确认优先级的扩展功能

## 5. 目标用户 Target Users

### 主要用户

第一版产品的核心用户是你这样的单人设计创作者。

这类用户希望借助 AI 更快完成从想法到视觉结果的生成过程，当前最关注的是 AI 生图，而不是多人协同或复杂管理能力。

### 当前阶段用户画像

- 以个人使用为主的设计师或视觉创作者
- 有明确的创作目标，希望快速把概念转成可见结果
- 愿意尝试 AI 工具，但不希望在多个工具之间频繁切换
- 更关注生成效率、可控性和结果质量，而不是复杂配置过程

### 用户使用目标

当前阶段，这类用户最希望完成的事情包括：

- 输入一个想法、主题或场景描述
- 通过 AI 快速生成可用图片
- 在生成结果不理想时快速调整提示词并再次生成
- 为后续扩展到 AI 视频生成打下素材和流程基础

### 用户主要痛点

围绕第一版 AI 生图能力，当前主要痛点包括：

- 从灵感到可执行提示词的转换成本高
- 不同 AI 工具分散，工作流不连贯
- 生成结果不稳定，需要频繁试错
- 难以快速沉淀和复用有效提示词、风格和素材
- 后续如果要继续做视频，前一步生成的图片和脚本缺少统一衔接

### 为什么先聚焦这类用户

当前产品由你自己使用，第一版更适合围绕最明确、最高频的使用场景收敛：

- 不先做多人协同
- 不先做复杂项目管理
- 优先把 AI 生图这一条主路径做顺

### 后续潜在用户

在第一版验证有效后，后续可以扩展到：

- 其他独立设计师或视觉创作者
- 需要快速产出概念图、参考图、情绪板的内容团队
- 有 AI 生图需求，并希望后续衔接 AI 视频能力的创作者

### 需要进一步确认的信息

- 用户最常生成的是海报、概念图、分镜图，还是短视频素材图
- 用户更在意生成速度、风格一致性，还是结果精细度
- 用户是否需要保存提示词模板、参考图和历史生成记录
- 第一版是否只服务单人使用，还是要提前考虑未来对外开放

## 6. 用户场景 User Scenarios

### 场景 A：直接输入 Prompt 生成图片

用户在无限画布中进行创作时，希望像在灵感板上工作一样自然。

他可以直接在画布中粘贴已有图片，也可以不放任何参考图，直接在输入框中输入 Prompt。点击发送后，系统基于输入内容生成一张新的图片，并将结果放回画布，方便继续查看、对比和迭代。

这个场景的核心目标是降低从“想法”到“第一张可见图片”的门槛。

### 场景 B：基于参考图修改或延展生成

用户先把一张参考图粘贴到无限画布中，然后在画布中选中这张图片。

当图片被选中后，输入框上方会明确显示当前选中的参考图，帮助用户确认这次生成将基于哪张图进行。用户随后输入 Prompt，并点击发送，系统根据“参考图 + 文本指令”生成结果。

在这个场景里，用户可能有两类典型目标：

- 对现有图片做修改、重绘或局部方向调整
- 基于现有图片的风格、构图或氛围，生成相似但新的图片

这个场景的核心目标是让用户能够围绕已有视觉素材继续扩展，而不是每次都从零开始。

### 场景 C：在画布中持续迭代和比较结果

用户不会只生成一次图片，而是会围绕同一个创意方向持续尝试多个 Prompt、多张参考图和多个生成结果。

无限画布对用户的价值，不只是展示最终结果，还包括：

- 保留输入与输出之间的上下文关系
- 支持把参考图、生成图、修改版本放在同一工作区中对比
- 让用户逐步收敛风格和结果，而不是频繁切换页面或工具

这个场景的核心目标是把 AI 生图过程变成一个连续创作流程，而不是孤立的一次次请求。

## 7. 核心需求 Requirements

### 核心能力

第一版产品需要优先支持以下核心能力：

- 提供一个可自由摆放内容的无限画布，作为主要创作工作区
- 支持用户通过复制粘贴的方式，直接把图片放入画布
- 提供文本输入框，用户输入 Prompt 后可直接发起 AI 生图
- 支持选中画布中的某张参考图，并将其带入输入框上方作为当前参考对象
- 支持基于“参考图 + Prompt”生成修改版图片，或生成相似风格的新图片
- 提供设置页面，用于配置第三方图片生成服务所需的 API 信息
- 支持用户在设置页中输入、保存和更新第三方 API Key 或相关接入参数
- 在未配置 API 信息时，对用户进行明确提示，避免生成流程直接失败
- 支持本地保存生成记录，至少包括生成后的图片和对应 Prompt
- 将每次生成的结果直接放回画布，便于继续比较、整理和迭代
- 针对高分辨率生成结果，支持在生成后派生缩略图用于画布展示
- 原图与缩略图需要区分用途：原图用于查看和导出，缩略图用于画布中的高频渲染
- 对生成中的状态、失败状态和可重试操作提供明确反馈

### 平台支持

第一版需要同时支持以下使用环境：

- PC 端 Web
- 移动端 Web

这意味着产品在交互和布局上不能只按桌面界面设计，而要从一开始考虑响应式结构和跨端可用性。

### 布局与交互要求

第一版需要具备基础自动布局能力，以适配不同屏幕尺寸和设备形态：

- 在 PC 端，优先保证画布操作空间充足，输入区和操作区清晰可见
- 在移动端，界面需要自动调整布局，避免画布、输入框和操作按钮相互遮挡
- 输入框、参考图预览、发送按钮等核心操作区域需要在不同分辨率下保持可访问
- 画布中的内容展示需要适应屏幕变化，避免核心内容超出可操作范围
- 当画布中存在较多图片节点时，优先使用缩略图参与展示，减少缩放、拖拽和多图同时渲染时的卡顿

### 体验要求

- 用户从输入想法到看到第一张生成结果的路径要尽量短
- 在有参考图和无参考图两种模式下，操作逻辑都要直观
- 选中参考图后，系统需要明确告知“当前生成将基于哪张图”
- 生成结果需要自然回到当前创作上下文中，而不是脱离画布单独展示
- 在失败、超时或结果不理想时，用户可以快速再次尝试
- 已生成内容需要在本地可恢复，避免关闭页面后创作上下文完全丢失
- 在大量图片同时存在的情况下，产品仍需保证基础可操作性，避免画布明显卡顿
- 第一版优先保证主路径顺畅，而不是追求过多高级配置项
- 设置页中的 API 配置流程要足够简单，让用户能够快速完成接入并开始生成

## 8. 范围 Scope

### In Scope

第一版明确包含以下范围：

- 无限画布作为主要创作空间
- 支持复制粘贴图片到画布
- 支持通过输入框上传图片
- 支持纯文本 Prompt 生图
- 支持参考图 + Prompt 生图
- 支持一次生成多张图片结果
- 支持重新生成和失败后重试
- 支持多张图片同时选中
- 支持拖拽、移动、整理和删除画布中的图片
- 支持基础撤回能力
- 支持设置页配置第三方图片生成服务的 API Key
- 支持接入多个第三方服务商
- 支持保存生成图片及其对应 Prompt
- 支持导出原图
- 支持对高分辨率原图生成缩略图，用于画布展示和性能优化
- 支持多个项目和多个画布
- 支持 PC Web 与移动端 Web（WAP）
- 支持基础响应式布局和自动布局适配
- 支持大量图片展示场景下的画布性能优化
- 支持 MacBook 触摸板操作，如双指缩放、拖拽画布

### Out of Scope

第一版明确不包含以下内容：

- 账号登录体系
- 云端同步
- 局部重绘或局部编辑能力
- 自动保存画布
- 与当前主流程无关的复杂高级编辑功能
- 以多人协同为核心的复杂权限和协作系统
- 超出第一版目标的非必要扩展能力
