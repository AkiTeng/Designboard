# 产品需求文档（PRD v1）

## 1. 产品定位

Designboard 是一个面向单人设计创作者的 AI 生图工具。

- 桌面端以无限画布为核心，承载整理、比较和迭代
- 桌面端画布除图片整理外，还要承载类似 Paper 的前端设计稿展示
- 移动端以流式列表为核心，承载轻量生成、查看和回看
- 用户可输入 Prompt、上传参考图、调用第三方模型生成图片，并将结果持续沉淀到本地项目与画布

一句话目标：让用户以尽可能短的路径，把一个想法迭代成一组可用图片，并保留完整创作上下文。

## 2. 产品原则

第一版只遵守以下原则：

- 优先缩短“想法 -> 首张结果图”的路径
- 优先保证围绕参考图的连续迭代体验
- 优先保证画布既能承载图片结果，也能承载设计稿式展示内容
- 优先保证本地项目、画布、图片和 Prompt 可恢复
- 优先保证桌面端在中等图片规模下交互流畅
- 不为了通用性和完整性引入过早抽象

这意味着第一版成功标准不是功能多，而是主路径顺、性能稳、数据能找回。

## 3. 问题定义

当前 AI 生图流程通常被拆散在多个工具中：

- Prompt 在一个地方写
- 参考图在另一个地方找
- 生成结果散落在多个目录或网页
- 多轮迭代后上下文难以保留
- 图片数量增多后，比较和筛选成本迅速上升

直接后果是：

- 从想法到结果的速度慢
- 从结果到下一轮迭代的成本高

## 4. 目标用户

第一版核心用户是单人设计创作者，尤其是频繁使用 AI 生图做视觉探索的人。

用户特征：

- 核心目标是快速产出图片，而不是管理复杂工程
- 需要在 Prompt、参考图、结果图之间高频切换
- 接受 AI 试错，但不接受流程割裂和画布卡顿
- 关注结果质量、迭代效率和上下文连续性

## 5. 产品目标

第一版只追求 5 个结果：

- 用户可以快速发起 AI 生图
- 用户可以围绕参考图持续迭代
- 生成结果可以在本地持续沉淀并重新打开
- 用户可以在桌面画布中查看和整理前端设计稿式内容
- 桌面端在单画布约 50 张图规模下保持可接受的拖拽和缩放体验

## 6. 非目标

第一版明确不做：

- 账号系统
- 云端同步
- 多人协作
- 局部重绘或局部编辑
- 复杂图像编辑器能力
- 第一版可交付的 GLSL 或实时特效编辑器
- 面向通用平台的过度抽象

## 7. 核心使用场景

### 场景 A：直接生图

用户输入 Prompt 后直接发起生成，系统返回结果并放回当前工作上下文。

价值：尽快完成“想到一个画面 -> 看到第一批结果”。

### 场景 B：参考图生图

用户上传或选中参考图，在输入区域确认当前参考图后输入 Prompt 发起生成。

价值：围绕已有素材继续延展，而不是每次从零开始。

### 场景 C：持续迭代

用户围绕一个方向连续生成多轮结果，并在同一项目或画布内比较、筛选、整理。

价值：让创作过程连续，而不是把每次生成都变成孤立事件。

### 场景 D：设计稿展示与风格预演

用户在桌面画布中放入类似 Paper 的前端设计稿内容，与生成图片混排展示，用于快速预演视觉方向、版式和氛围。

价值：让画布不仅是结果收纳区，也是可直接沟通视觉方案的展示空间。

## 8. 功能范围

### 8.1 In Scope

- 多项目、多画布
- 桌面端无限画布
- 移动端流式图片列表
- Prompt 输入
- 上传参考图
- 复制粘贴图片到桌面画布
- 纯文本生图
- 参考图 + Prompt 生图
- 一次生成多张图
- 失败后重试和再次生成
- 多图选中、拖拽、删除、撤回
- 多服务商 API Key 配置
- 本地文件持久化
- 原图与缩略图双资源策略
- 删除到回收桶
- 原图导出
- 桌面端支持设计稿式节点展示，可与图片节点混排
- 为后续 GLSL / Shader / 动态视觉效果预留扩展位

### 8.2 Out of Scope

- 登录和账号体系
- 跨设备同步
- 团队协作和权限系统
- 局部编辑能力
- 自动保存画布
- 移动端画布拖拽和缩放
- 非必要的高级参数控制面板
- 第一版不交付 GLSL 实时编辑、Shader 调参面板和特效时间轴

## 9. 需求拆解

### 9.1 输入模块

需要支持：

- 文本 Prompt 输入
- 上传参考图
- 复制粘贴图片到桌面画布
- 在发起生成前明确当前参考图上下文

### 9.2 生成模块

需要支持：

- 纯文本生图
- 参考图 + Prompt 生图
- 一次返回多张结果图
- 失败重试
- 基于同一输入再次生成

### 9.3 桌面画布模块

需要支持：

- 无限画布
- 图片节点与设计稿节点共存
- 多图同时选中
- 拖拽和整理
- 删除
- 撤回
- 触控板双指缩放和拖动画布
- 设计稿节点可展示排版、配色、图片、基础装饰效果
- 节点模型需为后续接入 GLSL、Shader 或动态效果保留扩展字段

### 9.4 移动端模块

第一版只保留：

- 流式图片列表
- 底部输入框
- 上传参考图
- 点击图片全屏查看
- 发起轻量生成

### 9.5 存储模块

需要支持：

- 本地文件保存
- 多项目
- 多画布
- 原图保存
- 缩略图保存
- Prompt 保存
- 删除图片进入回收桶
- 重新打开之前保存的项目和画布

### 9.6 性能模块

第一版必须满足：

- 原图与缩略图分离存储
- 画布默认优先显示缩略图
- 图片数量增加时优先保证拖拽和缩放流畅
- 设计稿节点渲染不能显著拖慢画布缩放、拖拽和首屏加载

## 10. 关键约束

- 桌面端是主创作场景，移动端是补充场景
- 第一版主要面向单人使用，不考虑多端协同一致性问题
- 图片可能为大图或 4K 图，不能以原图直接驱动画布渲染
- 第三方模型能力存在差异，产品层不能假设服务商行为完全一致
- 画布节点模型不能只为静态图片设计，需要兼容后续设计稿节点与特效节点扩展
- 第一版可以只交付静态设计稿展示，但技术方案不能阻塞后续接入 GLSL 等 GPU 效果

## 11. 风险

- 第三方服务商差异会提高接入和维护成本
- 本地文件方案在目标运行环境中的权限和兼容性需要提前验证
- 缩略图尺寸或压缩策略不合理时，画布性能仍可能失控
- 多项目、多画布文件结构设计不当会放大后续维护成本
- 如果节点模型只围绕图片设计，后续补设计稿与特效能力的返工成本会很高
- 如果渲染方案过早绑死在 DOM 结构上，后续接入高性能视觉效果会受限

## 12. 成功标准

满足以下条件即可认为第一版达标：

- 用户可以从新建项目到生成第一张图，完整顺畅地走通主路径
- 用户可以基于参考图继续生成，并清楚理解当前输入上下文
- 用户关闭应用后再次打开，仍能恢复本地项目和画布
- 用户可以在画布中稳定查看设计稿式内容，并与图片结果共同编排
- 单画布约 50 张图时，拖拽和缩放体验仍可接受
- 用户可以导出原图，不丢失高质量资源

## 13. 验收标准

- 桌面端主路径可用：新建项目、输入 Prompt、上传参考图、生成结果、整理图片
- 移动端核心路径可用：查看历史结果、上传参考图、发起轻量生成
- 本地文件结构可保存并重新打开
- 多服务商 API Key 可配置并生效
- 图片删除进入回收桶
- 原图与缩略图分离策略生效
- 设计稿节点可保存、重新打开并保持基础展示一致
- 文档明确标注 GLSL / 特效能力为后续版本能力，但框架已预留
- 文档内容足以支撑下一步技术方案设计与开发任务拆解

## 14. 待确认问题

1. 第一版优先接入哪些第三方服务商？
2. 本地文件方案的目标运行环境和权限边界是什么？
3. 缩略图的目标尺寸、压缩格式和生成时机如何定义？
4. 多图生成时每次默认返回几张图最合适？
5. 移动端是否需要支持选择历史参考图，而不只是当前上传图？
6. 设计稿节点第一版支持到什么粒度，是静态展示卡片、富布局容器，还是可导入 HTML/CSS 结果？
7. 后续 GLSL / 特效能力的目标场景是什么，是背景氛围、组件动效，还是完整 shader 画面？
